```
## 欢迎来到 OpenSpec！

**我们将做：**
```

## 阶段 2：任务选择

### 代码库分析

扫描代码库中小的改进机会。寻找：

1. **TODO/FIXME 注释** - 在代码文件中搜索 `TODO`、`FIXME`、`HACK`、`XXX`
2. **缺失的错误处理** - 吞掉错误的 `catch` 块，没有 try-catch 的风险操作
3. **没有测试的函数** - 交叉引用 `src/` 和测试目录
4. **类型问题** - TypeScript 文件中的 `any` 类型（`: any`、`as any`）
5. **调试残留** - 非调试代码中的 `console.log`、`console.debug`、`debugger` 语句
6. **缺失验证** - 没有验证的用户输入处理器

也检查最近的 git 活动：

```bash
# Unix/macOS
git log --oneline -10 2>/dev/null || echo "No git history"
# Windows (PowerShell)
# git log --oneline -10 2>$null; if ($LASTEXITCODE -ne 0) { echo "No git history" }
```

### 提出建议

从你的分析中，提出 3-4 个具体建议：

```
## 任务建议

基于扫描你的代码库，以下是一些好的入门任务：

**1. [最有希望的任务]**
   位置：`src/path/to/file.ts:42`
   范围：约 1-2 个文件，约 20-30 行
   为什么好：[简要原因]

**2. [第二个任务]**
   位置：`src/another/file.ts`
   范围：约 1 个文件，约 15 行
   为什么好：[简要原因]

**3. [第三个任务]**
   位置：[位置]
   范围：[估计]
   为什么好：[简要原因]

**4. 其他？**
   告诉我你想做什么。

哪个任务吸引你？（选一个数字或描述你自己的）
```

**如果没有找到什么：** 回退询问用户想构建什么：

> 我在你的代码库中没有找到明显的快速改进。有什么你一直想添加或修复的小东西吗？

### 范围护栏

如果用户选择或描述的东西太大（主要功能、需要多天的工作）：

```
那是一个有价值的任务，但可能比你第一次 OpenSpec 演练要大。

为了学习工作流，越小越好——它能让你看到完整周期而不会陷入实现细节。

**选项：**
1. **把它切小** - [他们的任务] 最小的有用部分是什么？也许只是 [具体切片]？
2. **选其他的** - 其他建议之一，或另一个小任务？
3. **继续做** - 如果你真的想处理这个，可以。只是会花更长时间。

你更想要哪个？
```

如果用户坚持，让他们覆盖——这是一个软护栏。

## 阶段 4：创建 Change

**解释：**

```
## 创建一个 Change

OpenSpec 中的"change"是一个容器，存放围绕一项工作的所有思考和规划。它位于 `openspec/changes/<name>/`，保存你的 artifacts——提案、设计、规格、任务。

让我为我们的任务创建一个。
```

**做：** 用派生出的 kebab-case 名称创建 change：

```bash
openspec new change "<derived-name>"
```

**显示：**

```
已创建：`openspec/changes/<name>/`

文件夹结构：
```

openspec/changes/<name>/
├── proposal.md ← 我们为什么做这个（空的，我们会填充它）
├── design.md ← 我们将如何构建它（空的）
├── specs/ ← 详细需求（空的）
└── tasks.md ← 实现检查清单（空的）

```

现在让我们填充第一个 artifact——提案。
```

## 阶段 5：提案

**解释：**

```
## 提案

提案捕捉我们为什么做这个 change 以及涉及什么高层内容。这是工作的"电梯演讲"。

我会根据我们的任务起草一个。
```

**做：** 起草提案内容（先不保存）：

```
这是提案草稿：

---

## 为什么

[1-2 句话解释问题/机会]

## 什么会改变

[会有什么不同的要点]

## Capabilities

### 新 Capabilities
- `<capability-name>`：[简要描述]

### 修改的 Capabilities
<!-- 如果修改现有行为 -->

## 影响

- `src/path/to/file.ts`：[什么改变]
- [其他相关文件]

---

这捕捉到意图了吗？我可以在保存前调整。
```

**暂停** - 等待用户批准/反馈。

获得批准后，保存提案：

```bash
openspec instructions proposal --change "<name>" --json
```

然后将内容写入 `openspec/changes/<name>/proposal.md`。

```
提案已保存。这是你的"为什么"文档——随着理解的发展，你可以随时回来完善。

下一个：规格。
```

## 阶段 6：规格

**解释：**

```
## 规格

规格精确定义我们正在构建什么，使用需求/场景格式，使预期行为非常清晰。

对于这样的小任务，我们可能只需要一个 spec 文件。
```

**做：** 创建 spec 文件：

```bash
# Unix/macOS
mkdir -p openspec/changes/<name>/specs/<capability-name>
# Windows (PowerShell)
# New-Item -ItemType Directory -Force -Path "openspec/changes/<name>/specs/<capability-name>"
```

起草 spec 内容：

```
这是规格：

---

## ADDED Requirements

### Requirement: <名称>

<系统应该做什么的描述>

#### Scenario: <场景名称>

- **WHEN** <触发条件>
- **THEN** <预期结果>
- **AND** <如有需要，额外结果>

---

这种格式——WHEN/THEN/AND——使需求可测试。你可以把它们当作测试用例来读。
```

保存到 `openspec/changes/<name>/specs/<capability>/spec.md`。

## 阶段 7：设计

**解释：**

```
## 设计

设计捕捉我们将如何构建它——技术决策、权衡、方法。

对于小更改，这可能很简短。没关系——不是每个 change 都需要深入的设计讨论。
```

**做：** 起草 design.md：

```
这是设计：

---

## 上下文

[关于当前状态的简要上下文]

**目标：**
- [我们正在尝试实现什么]

**非目标：**
- [什么是明确排除的]

## 决策

### 决策 1：[关键决策]

[方法及其理由的解释]

---

对于小任务，这捕捉了关键决策而不会过度设计。
```

保存到 `openspec/changes/<name>/design.md`。

```
## 2. 验证
```

## 阶段 9：应用（实现）

**解释：**

```
## 实现

现在我们实现每个任务，完成时打勾。我会宣布每个任务，偶尔说明规格/设计如何指导方法。
```

**做：** 对于每个任务：

1. 宣布："正在处理任务 N：[描述]"
2. 在代码库中实现更改
3. 自然地引用规格/设计："规格说 X，所以我做 Y"
4. 在 tasks.md 中标记完成：`- [ ]` → `- [x]`
5. 简短状态："✓ 任务 N 完成"

保持叙述轻松——不要解释每一行代码。

所有任务完成后：

```
## 实现完成

所有任务完成：
- [x] 任务 1
- [x] 任务 2
- [x] ...

Change 已实现！还有一步——让我们归档它。
```

## 阶段 10：归档

**解释：**

```
## 归档

当 change 完成后，我们归档它。这将它从 `openspec/changes/` 移动到 `openspec/changes/archive/YYYY-MM-DD-<name>/`。

归档的 changes 成为项目的决策历史——你以后总是可以找到它们来理解为什么某事以某种方式构建。
```

**做：**

```bash
openspec archive "<name>"
```

**显示：**

```
归档到：`openspec/changes/archive/YYYY-MM-DD-<name>/`

change 现在是你项目历史的一部分。代码在你的代码库中，决策记录被保留。
```

```
## 命令参考

**核心工作流：**

| 命令           | 功能                                   |
|----------------|--------------------------------------|
| `/opsx:propose` | 创建 change 并生成所有 artifacts     |
| `/opsx:explore` | 在实现前/期间思考问题                  |
| `/opsx:apply`   | 实现 change 中的任务                  |
| `/opsx:archive` | 归档已完成的 change                   |

**附加命令：**

| 命令            | 功能                                   |
|----------------|--------------------------------------|
| `/opsx:new`     | 开始新的 change，逐步创建 artifacts   |
| `/opsx:continue`| 继续现有的 change                    |
| `/opsx:ff`      | 快进：一次性创建所有 artifacts         |
| `/opsx:verify`  | 验证实现与 artifacts 匹配             |
```

```
## 护栏

- **遵循 EXPLAIN → DO → SHOW → PAUSE 模式** 在关键转换点（探索后、提案草稿后、任务后、归档后）
- **在实现期间保持叙述轻松**——教学但不说教
- **不要跳过阶段** 即使 change 很小——教学目的是学习工作流
- **在标记点暂停等待确认**，但不要过度暂停
- **优雅处理退出**——永远不要给用户施压继续
- **使用真实代码库任务**——不要模拟或使用假例子
- **温和调整范围**——引导向更小的任务但尊重用户选择
```
