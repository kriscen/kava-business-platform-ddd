## 用户输入

```text
$ARGUMENTS
```

如果用户输入不为空，你**必须**在继续之前考虑用户输入。

## 执行前检查

**检查扩展钩子（规格前）**：
- 检查项目根目录是否存在 `.specify/extensions.yml`。
- 如存在，读取并查找 `hooks.before_specify` 键下的条目
- 如YAML无法解析或无效，静默跳过钩子检查并正常继续
- 过滤掉 `enabled` 显式为 `false` 的钩子。将无 `enabled` 字段的钩子视为默认启用。
- 对于每个剩余钩子，**不要**尝试解释或评估钩子 `condition` 表达式：
  - 如钩子无 `condition` 字段，或为null/空，视为钩子可执行
  - 如钩子定义非空 `condition`，跳过钩子并将条件评估留给HookExecutor实施
- 对于每个可执行钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：
    ```
    ## 扩展钩子

    **可选前置钩子**：{extension}
    命令：`/{command}`
    描述：{description}

    提示：{prompt}
    执行：`/{command}`
    ```
  - **强制钩子**（`optional: false`）：
    ```
    ## 扩展钩子

    **自动前置钩子**：{extension}
    执行：`/{command}`
    EXECUTE_COMMAND: {command}

    等待钩子命令结果后再继续概要。
    ```
- 如无注册钩子或 `.specify/extensions.yml` 不存在，静默跳过

## 概要

用户在触发消息中 `/speckit.specify` 后输入的文本**就是**功能描述。假设你在此对话中始终有它可用，即使 `$ARGUMENTS` 下面显示字面值。除非用户提供空命令，不要要求用户重复。

给定该功能描述，执行以下：

1. **生成简洁短名称**（2-4词）用于分支：
   - 分析功能描述并提取最有意义的关键词
   - 创建捕获功能本质的2-4词短名称
   - 尽可能使用动作-名词格式（如，"add-user-auth"、"fix-payment-bug"）
   - 保留技术术语和缩写（OAuth2、API、JWT等）
   - 保持简洁但描述性足以一目了然理解功能
   - 示例：
     - "I want to add user authentication" → "user-auth"
     - "Implement OAuth2 integration for the API" → "oauth2-api-integration"
     - "Create a dashboard for analytics" → "analytics-dashboard"
     - "Fix payment processing timeout bug" → "fix-payment-timeout"

2. **通过运行带 `--short-name`（和 `--json`）的脚本创建功能分支**。在顺序模式下，**不**传递 `--number`——脚本自动检测下一个可用数字。在时间戳模式下，脚本自动生成 `YYYYMMDD-HHMMSS` 前缀：

   **分支编号模式**：运行脚本前，检查 `.specify/init-options.json` 是否存在并读取 `branch_numbering` 值。
   - 如 `"timestamp"`，添加 `--timestamp`（Bash）或 `-Timestamp`（PowerShell）到脚本调用
   - 如 `"sequential"` 或缺失，不添加额外标志（默认行为）

   - Bash示例：`.specify/scripts/bash/create-new-feature.sh "$ARGUMENTS" --json --short-name "user-auth" "Add user authentication"`
   - Bash（时间戳）：`.specify/scripts/bash/create-new-feature.sh "$ARGUMENTS" --json --timestamp --short-name "user-auth" "Add user authentication"`
   - PowerShell示例：`.specify/scripts/bash/create-new-feature.sh "$ARGUMENTS" -Json -ShortName "user-auth" "Add user authentication"`
   - PowerShell（时间戳）：`.specify/scripts/bash/create-new-feature.sh "$ARGUMENTS" -Json -Timestamp -ShortName "user-auth" "Add user authentication"`

   **重要**：
   - **不**传递 `--number`——脚本自动确定正确的下一个数字
   - 始终包含JSON标志（Bash用 `--json`，PowerShell用 `-Json`）以便输出可靠解析
   - 每个功能只能运行此脚本一次
   - JSON在终端作为输出提供——始终引用它获取你要找的实际内容
   - JSON输出将包含 BRANCH_NAME 和 SPEC_FILE 路径
   - 对于参数中的单引号，如 "I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或者尽可能使用双引号："I'm Groot"）

3. 加载 `.specify/templates/spec-template.md` 了解必需章节。

4. 遵循此执行流程：

    1. 从输入解析用户描述
       如空：ERROR "未提供功能描述"
    2. 从描述提取关键概念
       识别：角色、动作、数据、约束
    3. 对于不清楚方面：
       - 基于上下文和行业标准做出知情猜测
       - 仅在以下情况用 [需要澄清：具体问题] 标记：
         - 选择显著影响功能范围或用户体验
         - 存在多个合理解释具有不同含义
         - 无合理默认值
       - **限制：最多总共3个 [需要澄清] 标记**
       - 按影响优先澄清：范围 > 安全/隐私 > 用户体验 > 技术细节
    4. 填充用户场景与测试章节
       如无清晰用户流程：ERROR "无法确定用户场景"
    5. 生成功能需求
       每个需求必须可测试
       对未指定细节使用合理默认值（在假设章节文档假设）
    6. 定义成功标准
       创建可衡量的、技术无关的结果
       包含定量指标（时间、性能、量）和定性度量（用户满意度、任务完成）
       每个标准必须可验证无需实施细节
    7. 识别关键实体（如涉及数据）
    8. 返回：SUCCESS（规格准备好计划）

5. 使用模板结构将规格写入 SPEC_FILE，用从功能描述（参数）推导的具体细节替换占位符，同时保持章节顺序和标题。

6. **规格质量验证**：写入初始规格后，按质量标准验证：

   a. **创建规格质量检查清单**：使用检查清单模板结构在 `FEATURE_DIR/checklists/requirements.md` 生成检查清单文件，包含以下验证项目：

      ```markdown
      # 规格质量检查清单：[功能名称]

      **目的**：在进入计划前验证规格完整性和质量
      **创建**：[日期]
      **功能**：[链接到 spec.md]

      ## 内容质量

      - [ ] 无实施细节（语言、框架、API）
      - [ ] 聚焦用户价值和业务需求
      - [ ] 为非技术利益相关者编写
      - [ ] 所有强制章节已完成

      ## 需求完整性

      - [ ] 无 [需要澄清] 标记残留
      - [ ] 需求可测试且明确
      - [ ] 成功标准可衡量
      - [ ] 成功标准技术无关（无实施细节）
      - [ ] 所有验收场景已定义
      - [ ] 边缘情况已识别
      - [ ] 范围清晰界定
      - [ ] 依赖和假设已识别

      ## 功能就绪

      - [ ] 所有功能需求有清晰验收标准
      - [ ] 用户场景覆盖主要流程
      - [ ] 功能满足成功标准中定义的可衡量结果
      - [ ] 无实施细节泄露到规格中

      ## 备注

      - 标记未完成的项目需要在 `/speckit.clarify` 或 `/speckit.plan` 前更新规格
      ```

   b. **运行验证检查**：按每个检查清单项目审阅规格：
      - 对于每个项目，确定通过或失败
      - 文档发现的具体问题（引用相关规格章节）

   c. **处理验证结果**：

      - **如所有项目通过**：标记检查清单完成并继续步骤7

      - **如项目失败（排除 [需要澄清]）**：
        1. 列出失败项目和具体问题
        2. 更新规格解决每个问题
        3. 重新运行验证直到所有项目通过（最多3轮）
        4. 如3轮后仍失败，在检查清单备注中记录剩余问题并警告用户

      - **如 [需要澄清] 标记残留**：
        1. 从规格提取所有 [需要澄清: ...] 标记
        2. **限制检查**：如超过3个标记，仅保留3个最关键的（按范围/安全/UX影响）并对其余做出知情猜测
        3. 对于每个需要澄清的（最多3个），以此格式向用户展示选项：

           ```markdown
           ## 问题 [N]：[主题]

           **上下文**：[引用相关规格章节]

           **需要了解**：[来自需要澄清标记的具体问题]

           **建议答案**：

           | 选项 | 答案 | 含义 |
           |------|------|------|
           | A    | [第一个建议答案] | [这对功能意味着什么] |
           | B    | [第二个建议答案] | [这对功能意味着什么] |
           | C    | [第三个建议答案] | [这对功能意味着什么] |
           | 自定义 | 提供你自己的答案 | [解释如何提供自定义输入] |

           **你的选择**：_[等待用户响应]_
           ```

        4. **关键 - 表格格式**：确保markdown表格正确格式化：
           - 使用一致间距，管道对齐
           - 每个单元格内容周围应有空格：`| 内容 |` 不是 `|内容|`
           - 标题分隔符至少有3个破折号：`|--------|`
           - 测试表格在markdown预览中正确渲染
        5. 顺序编号问题（Q1、Q2、Q3 - 总共最多3个）
        6. 在等待响应前一起展示所有问题
        7. 等待用户用所有问题的选择响应（如，"Q1: A, Q2: 自定义 - [详情], Q3: B"）
        8. 用用户选择或提供的答案替换每个 [需要澄清] 标记更新规格
        9. 所有澄清解决后重新运行验证

   d. **更新检查清单**：每轮验证后，用当前通过/失败状态更新检查清单文件

7. 报告完成，包含分支名称、规格文件路径、检查清单结果和下一阶段就绪状态（`/speckit.clarify` 或 `/speckit.plan`）。

8. **检查扩展钩子**：报告完成后，检查项目根目录是否存在 `.specify/extensions.yml`。
   - 如存在，读取并查找 `hooks.after_specify` 键下的条目
   - 如YAML无法解析或无效，静默跳过钩子检查并正常继续
   - 过滤掉 `enabled` 显式为 `false` 的钩子。将无 `enabled` 字段的钩子视为默认启用。
   - 对于每个剩余钩子，**不要**尝试解释或评估钩子 `condition` 表达式：
     - 如钩子无 `condition` 字段，或为null/空，视为钩子可执行
     - 如钩子定义非空 `condition`，跳过钩子并将条件评估留给HookExecutor实施
   - 对于每个可执行钩子，根据其 `optional` 标志输出以下内容：
     - **可选钩子**（`optional: true`）：
       ```
       ## 扩展钩子

       **可选钩子**：{extension}
       命令：`/{command}`
       描述：{description}

       提示：{prompt}
       执行：`/{command}`
       ```
     - **强制钩子**（`optional: false`）：
       ```
       ## 扩展钩子

       **自动钩子**：{extension}
       执行：`/{command}`
       EXECUTE_COMMAND: {command}
       ```
   - 如无注册钩子或 `.specify/extensions.yml` 不存在，静默跳过

**注意**：脚本创建并检出新分支，在写入前初始化规格文件。

## 快速指南

- 聚焦用户**需要什么**和**为什么**。
- 避免**如何**实施（无技术栈、API、代码结构）。
- 为业务利益相关者编写，非开发者。
- **不要**创建嵌入在规格中的任何检查清单。那将是单独的命令。

### 章节要求

- **强制章节**：每个功能必须完成
- **可选章节**：仅在与功能相关时包含
- 当章节不适用时，完全移除（不留"N/A"）

### AI生成时

从用户提示创建此规格时：

1. **做出知情猜测**：使用上下文、行业标准和常见模式填补缺口
2. **文档假设**：在假设章节记录合理默认值
3. **限制澄清**：最多3个 [需要澄清] 标记 - 仅用于关键决策：
   - 显著影响功能范围或用户体验
   - 有多个合理解释具有不同含义
   - 缺乏任何合理默认值
4. **优先澄清**：范围 > 安全/隐私 > 用户体验 > 技术细节
5. **像测试者思考**：每个模糊需求应失败"可测试且明确"检查清单项目
6. **常见需要澄清区域**（仅如无合理默认值）：
   - 功能范围和边界（包含/排除特定用例）
   - 用户类型和权限（如多个冲突解释可能）
   - 安全/合规需求（当法律/财务重要）

**合理默认值示例**（不要问这些）：

- 数据保留：领域的行业标准实践
- 性能目标：标准Web/移动应用期望（除非指定）
- 错误处理：带适当回退的用户友好消息
- 认证方法：Web应用的标准基于会话或OAuth2
- 集成模式：使用项目适合的模式（Web服务用REST/GraphQL，库用函数调用，工具用CLI参数等）

### 成功标准指南

成功标准必须：

1. **可衡量**：包含具体指标（时间、百分比、计数、比率）
2. **技术无关**：不提及框架、语言、数据库或工具
3. **用户聚焦**：从用户/业务视角描述结果，非系统内部
4. **可验证**：可测试/验证无需了解实施细节

**好示例**：

- "用户可在3分钟内完成结账"
- "系统支持10,000并发用户"
- "95%的搜索在1秒内返回结果"
- "任务完成率提高40%"

**坏示例**（实施聚焦）：

- "API响应时间低于200ms"（太技术，用"用户即时看到结果"）
- "数据库可处理1000 TPS"（实施细节，用用户面向指标）
- "React组件高效渲染"（框架特定）
- "Redis缓存命中率高于80%"（技术特定）
