## 用户输入

```text
$ARGUMENTS
```

如果用户输入不为空，你**必须**在继续之前考虑用户输入。

## 概要

目标：检测并减少活动功能规格中的歧义或缺失决策点，并将澄清直接记录在规格文件中。

注意：此澄清工作流预期在调用 `/speckit.plan` 之前运行（并完成）。如果用户明确表示跳过澄清（如，探索性冲刺），你可以继续，但必须警告下游返工风险增加。

执行步骤：

1. 从仓库根目录运行 `.specify/scripts/bash/check-prerequisites.sh --json --paths-only` **一次**（组合 `--json --paths-only` 模式 / `-Json -PathsOnly`）。解析最小JSON载荷字段：
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - （可选捕获 `IMPL_PLAN`、`TASKS` 用于后续链式流程。）
   - 如果JSON解析失败，中止并指示用户重新运行 `/speckit.specify` 或验证功能分支环境。
   - 对于参数中的单引号，如 "I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或者尽可能使用双引号："I'm Groot"）。

2. 加载当前规格文件。使用此分类法执行结构化的歧义与覆盖扫描。对于每个类别，标记状态：清晰 / 部分 / 缺失。生成用于优先排序的内部覆盖图（除非不提问否则不输出原始图）。

   功能范围与行为：
   - 核心用户目标与成功标准
   - 显式的范围外声明
   - 用户角色/角色区分

   领域与数据模型：
   - 实体、属性、关系
   - 身份与唯一性规则
   - 生命周期/状态转换
   - 数据量/规模假设

   交互与UX流程：
   - 关键用户旅程/序列
   - 错误/空/加载状态
   - 无障碍或本地化说明

   非功能性质量属性：
   - 性能（延迟、吞吐目标）
   - 可扩展性（横向/纵向、限制）
   - 可靠性与可用性（正常运行时间、恢复期望）
   - 可观测性（日志、指标、追踪信号）
   - 安全与隐私（认证/授权、数据保护、威胁假设）
   - 合规/监管约束（如有）

   集成与外部依赖：
   - 外部服务/API和失败模式
   - 数据导入/导出格式
   - 协议/版本假设

   边缘情况与失败处理：
   - 负面场景
   - 速率限制/节流
   - 冲突解决（如，并发编辑）

   约束与权衡：
   - 技术约束（语言、存储、托管）
   - 显式权衡或拒绝的替代方案

   术语与一致性：
   - 规范术语表术语
   - 避免的同义词/弃用术语

   完成信号：
   - 验收标准可测试性
   - 可衡量的完成定义风格指标

   其他/占位符：
   - TODO标记/未解决决策
   - 缺乏量化的模糊形容词（"健壮"、"直观"）

   对于每个部分或缺失状态的类别，添加候选问题机会，除非：
   - 澄清不会实质性改变实施或验证策略
   - 信息更适合推迟到计划阶段（内部记录）

3. 内部生成优先排序的候选澄清问题队列（最多5个）。不要一次性全部输出。应用这些约束：
    - 整个会话最多5个问题。
    - 每个问题必须可以用以下方式回答：
       - 简短多选选择（2–5个互斥选项），或
       - 一词/短语答案（显式限制："用<=5个词回答"）。
    - 仅包含答案实质性影响架构、数据建模、任务分解、测试设计、UX行为、运维准备或合规验证的问题。
    - 确保类别覆盖平衡：尝试先覆盖最高影响的未解决类别；避免问两个低影响问题当单个高影响区域（如，安全态势）未解决。
    - 排除已回答的问题、琐碎的风格偏好或计划级执行细节（除非阻塞正确性）。
    - 优先减少下游返工风险或防止不对齐验收测试的澄清。
    - 如果超过5个类别仍未解决，按（影响 * 不确定性）启发式选择前5个。

4. 顺序提问循环（交互式）：
    - 每次呈现**恰好一个问题**。
    - 对于多选问题：
       - **分析所有选项**并基于以下确定**最合适的选项**：
          - 项目类型的最佳实践
          - 类似实施的常见模式
          - 风险降低（安全、性能、可维护性）
          - 与规格中可见的任何显式项目目标或约束对齐
       - 在顶部**显著呈现你的推荐选项**，附带清晰理由（1-2句解释为何这是最佳选择）。
       - 格式为：`**推荐：** 选项 [X] - <理由>`
       - 然后以Markdown表格渲染所有选项：

       | 选项 | 描述 |
       |------|------|
       | A | <选项A描述> |
       | B | <选项B描述> |
       | C | <选项C描述>（需要时添加D/E，最多5个） |
       | 简短 | 提供不同的简短答案（<=5词）（仅当自由格式替代合适时包含） |

       - 表格后添加：`你可以回复选项字母（如，"A"），说"yes"或"推荐"接受推荐，或提供你自己的简短答案。`
    - 对于简答风格（无有意义的离散选项）：
       - 基于最佳实践和上下文提供你的**建议答案**。
       - 格式为：`**建议：** <你的建议答案> - <简短理由>`
       - 然后输出：`格式：简短答案（<=5词）。你可以说"yes"或"建议"接受建议，或提供你自己的答案。`
    - 用户回答后：
       - 如果用户回复"yes"、"推荐"或"建议"，使用你之前陈述的推荐/建议作为答案。
       - 否则，验证答案映射到一个选项或符合<=5词约束。
       - 如果模糊，请求快速澄清（计数仍属于同一问题；不前进）。
       - 一旦满意，在工作内存中记录（尚未写入磁盘）并进入下一个排队问题。
    - 当以下情况停止提问：
       - 所有关键歧义早期解决（剩余排队项目变得不必要），或
       - 用户发出完成信号（"done"、"good"、"no more"），或
       - 你达到5个已问问题。
    - 从不提前揭示未来排队问题。
    - 如果开始时无有效问题存在，立即报告无关键歧义。

5. 每个接受答案后的整合（增量更新方法）：
    - 在内存中维护规格表示（开始时加载一次）加上原始文件内容。
    - 对于本会话中第一个整合答案：
       - 确保存在 `## 澄清` 章节（如果缺失，按规格模板在最高级上下/概述章节后创建）。
       - 在其下创建（如果不存在）`### 会话 YYYY-MM-DD` 子标题为今天。
    - 接受后立即追加要点行：`- 问: <问题> → 答: <最终答案>`。
    - 然后立即将澄清应用到最合适的章节：
       - 功能歧义 → 更新或在功能需求中添加要点。
       - 用户交互/角色区分 → 用澄清的角色、约束或场景更新用户故事或角色子章节（如果存在）。
       - 数据形状/实体 → 更新数据模型（添加字段、类型、关系）保持排序；简洁记录添加的约束。
       - 非功能性约束 → 在成功标准 > 可衡量结果中添加/修改可衡量标准（将模糊形容词转换为指标或显式目标）。
       - 边缘情况/负面流程 → 在边缘情况/错误处理下添加新要点（或如果模板为其提供占位符则创建此子章节）。
       - 术语冲突 → 在规格中规范术语；仅在必要时保留原始，添加 `(曾称为"X")` 一次。
    - 如果澄清使早期模糊陈述无效，替换该陈述而非复制；不留过时矛盾文本。
    - 每次整合后保存规格文件以最小化上下文丢失风险（原子覆盖）。
    - 保持格式：不重排无关章节；保持标题层次完整。
    - 保持每个插入的澄清最小且可测试（避免叙述漂移）。

6. 验证（每次写入后执行加最终通过）：
   - 澄清会话恰好包含每个接受答案一个要点（无重复）。
   - 总共询问（接受）问题 ≤ 5。
   - 更新章节不包含新答案本应解决的遗留模糊占位符。
   - 无矛盾早期陈述残留（扫描已移除的现在无效替代选择）。
   - Markdown结构有效；仅允许新标题：`## 澄清`、`### 会话 YYYY-MM-DD`。
   - 术语一致性：所有更新章节使用相同规范术语。

7. 将更新规格写回 `FEATURE_SPEC`。

8. 报告完成（提问循环结束或提前终止后）：
   - 询问并回答的问题数量。
   - 更新规格的路径。
   - 涉及章节（列出名称）。
   - 覆盖摘要表列出每个分类法类别及其状态：已解决（曾为部分/缺失并已处理）、推迟（超出问题配额或更适合计划）、清晰（已足够）、待定（仍为部分/缺失但低影响）。
   - 如有任何待定或推迟，建议是否继续 `/speckit.plan` 或稍后计划后再运行 `/speckit.clarify`。
   - 建议下一步命令。

行为规则：

- 如果无有意义歧义发现（或所有潜在问题都低影响），响应："未检测到值得正式澄清的关键歧义。"并建议继续。
- 如果规格文件缺失，指示用户先运行 `/speckit.specify`（在此不创建新规格）。
- 从不超过总共5个已问问题（单个问题的澄清重试不计为新问题）。
- 避免推测性技术栈问题除非缺失阻塞功能清晰度。
- 尊重用户提前终止信号（"stop"、"done"、"proceed"）。
- 如果因完全覆盖无问题，输出紧凑覆盖摘要（所有类别清晰）然后建议前进。
- 如果配额达到但有高影响类别未解决，在推迟下显式标记并附带理由。
