# 06-modules — 模块级文档

## 目录与项目模块的对应关系

本目录下的子目录与项目 Maven 子模块一一对应，目前包含：

| 目录名 | 模块类型 | 说明 |
|--------|----------|------|
| `kbpd-auth` | 功能模块 | OAuth2 认证授权服务 |
| `kbpd-common` | 公共模块 | 公共依赖、工具类、基础类型 |
| `kbpd-gateway` | 功能模块 | API 网关 |
| `kbpd-member` | 业务模块 | 会员服务 |
| `kbpd-upms` | 业务模块 | 用户权限管理服务 |

新增 Maven 子模块时，应在本目录下创建同名的子目录，并视模块类型填充相应文档。

## 文档产出规范

### 业务模块（`kbpd-member`、`kbpd-upms` 等）

业务模块需包含以下三类文档：

- **`overview.md`** — 模块全景：职责边界、领域划分、核心聚合、依赖关系、架构分层说明
- **`business-rules.md`** — 业务规则：领域约束、状态机、关键业务流程、权限/资金/数据完整性相关的规则描述
- **`api.md`** — 接口文档：当前模块对外暴露的接口（REST / Dubbo / 内部 API）。
  > 注：若接口数量较大，后续可考虑引入 Apifox MCP 规则以自动化维护，届时在此处补充相关说明。

### 公共模块 / 功能模块（`kbpd-common`、`kbpd-gateway`、`kbpd-auth` 等）

此类模块以一份 `overview.md` 为主即可，内容涵盖模块职责、核心配置、对外提供的公共能力或使用方式。

- 若公共模块有对外接口需要额外说明，可单独补充 `api.md`，非强制。
- 不强制要求 `business-rules.md`，因为不涉及领域业务规则。

## 维护提醒

- 目录结构变更（新增/删除/重命名模块文档）后，执行 `/update-docs-map` 同步 `docs/00-project-map.md`。
