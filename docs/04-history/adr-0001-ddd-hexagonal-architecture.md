# 2024-01-15 采用 DDD + 六边形架构

## 状态
已采纳

## 背景

项目定位为**企业级业务平台**，预期业务复杂度高、团队规模大、需要长期维护演进。传统的三层架构（Controller-Service-DAO）在应对复杂业务逻辑时存在以下问题：

- **业务逻辑泄漏**：Service 层容易变成事务脚本，业务规则散落各处
- **依赖方向混乱**：业务代码直接依赖数据库、框架，难以测试和替换
- **领域知识丢失**：贫血模型无法表达业务概念，团队沟通成本高

## 决策

采用**领域驱动设计（DDD）** + **六边形架构（端口与适配器）** 作为核心架构范式：

1. **严格分层**：adapter/infrastructure → application → domain，依赖方向必须向内
2. **Domain Layer 隔离**：核心业务逻辑不依赖任何外部框架（Spring、MyBatis 等）
3. **Ports & Adapters**：通过端口抽象外部依赖，应用层通过接口编程

```
adapter/infrastructure  →  application  →  domain
```

## 理由

1. **可测试性**：Domain 层无外部依赖，可直接单元测试业务规则
2. **业务表达力**：聚合、值对象、领域事件等模式让代码反映业务语言
3. **技术演进**：端口抽象使数据库、框架替换成本低，支持渐进式重构
4. **团队协作**：BC（bounded context）划分支持多团队并行开发

## 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 传统三层架构 | 简单直接 | 业务逻辑泄漏，难以应对复杂领域 |
| 整洁架构（Clean） | 层次清晰 | 对 DDD 概念支持较弱，边界定义模糊 |
| 事件溯源（Event Sourcing） | 可追溯性强 | 复杂度高，团队学习成本大 |

## 后果

- **正面**：业务代码内聚、可测试性强，长期维护成本降低
- **正面**：团队需要学习 DDD 概念，初期有一定学习曲线
- **约束**：Domain 层禁止依赖 Spring、MyBatis 等框架注解
- **约束**：所有业务规则必须沉淀在 Domain 层，不得泄漏到 Application 或 Adapter
