# 2024-02-15 跨服务一致性：Saga 模式

## 状态
已采纳

## 背景

微服务架构下，每个服务管理自己的数据库，跨服务的数据一致性无法通过传统分布式事务（2PC）解决：

- **跨库一致性需求**：创建用户（UPMS）后，需要自动创建会员档案（Member）
- **性能要求**：不能因为强一致性而阻塞整个系统
- **服务独立**：各服务应独立演进，不应被事务紧耦合

## 决策

采用 **Saga 模式**处理跨服务数据一致性：

```
┌─────────────────────────────────────────────────────────────────┐
│                     Saga: 创建用户 + 会员档案                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  Step 1: UPMS 创建用户 (本地事务)                                  │
│          ├─► 成功 ──► Step 2                                      │
│          └─► 失败 ──► Saga 结束（无补偿）                          │
│                                                                   │
│  Step 2: Member 创建会员档案 (远程调用)                             │
│          ├─► 成功 ──► Saga 完成                                   │
│          └─► 失败 ──► Step 3 补偿 (UPMS 删除用户)                  │
│                                                                   │
│  Step 3: UPMS 补偿操作 (删除已创建用户)                             │
│          ├─► 成功 ──► Saga 回滚完成                                │
│          └─► 失败 ──► 人工介入 / 告警                               │
└─────────────────────────────────────────────────────────────────┘
```

**实现方式**：

1. **领域事件驱动**：聚合根发布事件（如 `UserCreatedEvent`）
2. **MQ 解耦**：通过消息队列（如 RocketMQ）异步传递事件
3. **补偿事务**：下游失败时，上游服务执行补偿操作

## 理由

1. **松耦合**：服务间通过事件协作，不依赖直接调用
2. **高性能**：异步处理，不阻塞主流程
3. **可追溯**：领域事件记录了所有业务操作，便于审计
4. **最终一致**：适合业务允许短暂不一致的场景

## 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 2PC/XA 事务 | 强一致 | 性能差，无法跨异构数据库 |
| TCC 模式 | 强一致 | 实现复杂，补偿逻辑复杂 |
| 本地消息表 | 可靠 | 实现较重 |

## 后果

- **正面**：服务独立演进，不被事务耦合
- **正面**：高性能，适合高并发场景
- **约束**：业务必须接受最终一致性，不能要求实时强一致
- **约束**：补偿逻辑必须幂等实现
- **约束**：每个 Saga 步骤必须可补偿
