# 菜单与应用模型

本文档描述 Kava 平台的菜单属性体系、应用（App）概念，以及菜单按购买关系分发的完整设计。

## 背景

Kava 平台存在两类管理角色：**平台管理员**和**租户管理员**。平台以"家庭"为租户单位，向租户提供可购买的业务应用（App）。每个 App 由平台方开发，租户购买后可定制品牌信息（名称、图标等），通过独立 OAuth Client 接入，形成租户专属的 App/小程序实例。

菜单是管理后台的功能入口，需要根据角色类型和 App 购买状态动态呈现。

## 核心概念

### 三层模型

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   菜单 (Menu)                                               │
│   • 独立实体，有自己的 level (PLATFORM / TENANT)              │
│   • level 决定平台管理员看还是租户管理员看                     │
│   • 不直接属于某个 App                                       │
│                                                             │
│   应用 (App)                                                │
│   • 菜单的组合包                                             │
│   • kava-base 是系统内置包，所有租户自动拥有                   │
│   • 业务 App 通过购买获得                                    │
│                                                             │
│   App-菜单 (多对多)                                         │
│   • 一个菜单可被多个 App 复用                                │
│   • 一个 App 可包含多个菜单                                  │
│                                                             │
│   租户-App (购买关系)                                        │
│   • 租户拥有 kava-base + 已购业务 App                        │
│   • 所有 App 的菜单并集 = 该租户的可见菜单候选集              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 菜单 level

菜单只有两个 level 值：

| level | 含义 | 谁能看到 |
|-------|------|---------|
| PLATFORM | 平台管理级菜单 | 平台管理员 |
| TENANT | 租户级菜单 | 租户管理员（受 App 购买关系约束） |

**共享菜单的处理**：如果一个业务功能需要平台管理员和租户管理员都能看到但数据权限不同，拆成两条菜单记录（一个 PLATFORM，一个 TENANT），可指向相同的前端组件，后端根据用户身份返回不同范围的数据。

### 菜单分类

```
level=PLATFORM, app_id=NULL:
  平台管理菜单（kava-base 提供）
  ├── 租户管理
  ├── 应用管理（管理 App、推送给租户）
  ├── 平台用户管理
  └── 全局配置

level=TENANT, app_id=NULL:
  租户基础菜单（kava-base 提供，所有租户自动拥有）
  ├── 用户管理
  ├── 角色管理
  ├── 部门管理
  └── ...

level=TENANT, app_id=ledger:
  记账 App 专属菜单（购买后拥有）
  ├── 账本管理
  ├── 收支记录
  └── 统计报表

level=TENANT, app_id=calendar:
  日程 App 专属菜单（购买后拥有）
  ├── 日程管理
  ├── 提醒设置
  └── 日历视图
```

## 应用体系

### 系统应用 kava-base

Kava 平台内置的系统应用，创建租户时自动关联，不会过期：

- 包含平台管理所需的 PLATFORM 级菜单
- 包含租户基础管理所需的 TENANT 级菜单
- 所有租户自动拥有，无需购买

### 业务应用

由平台方开发的垂直业务应用，租户通过订阅购买：

- 每个 App 注册时声明自己包含的菜单集
- 可包含 PLATFORM 级菜单（如平台级数据看板）和 TENANT 级菜单（如业务管理）
- 同一个菜单可出现在多个 App 中（复用），购买时自动去重

### 菜单复用

```
kava-base       ledger          calendar
┌────────┐     ┌────────┐      ┌────────┐
│ id=10  │     │ id=20  │◄────►│ id=20  │  复用（会员管理）
│ id=11  │     │ id=30  │      │ id=21  │  复用（消息通知）
│ id=12  │     │ id=31  │      │ id=40  │
└────────┘     └────────┘      │ id=41  │
                                └────────┘
```

当租户同时购买记账和日程 App 时，id=20（会员管理）只出现一次。

## 分发规则

### 可见性链路

```
平台管理员：
  → 所有 level=PLATFORM 的菜单（不受 App 购买关系约束）

租户管理员：
  → 该租户拥有的所有 App 的菜单并集（自动去重）
  → 过滤 level=TENANT
  → 再按用户角色权限过滤
  → 构建最终菜单树
```

### 购买即生效

菜单不需要显式"推送"。菜单始终存在于 sys_menu 表中，通过 sys_tenant_app 的购买关系控制可见性——声明式而非命令式。

```
租户购买 App
  → sys_tenant_app 写入记录
  → 查询逻辑自动包含该 App 的菜单
  → 租户管理员在角色管理页面可看到并分配新菜单
```

### 订阅到期

```
租户 App 订阅到期
  → sys_tenant_app.status → EXPIRED
  → 查询逻辑排除 EXPIRED 的 App
  → 对应菜单自动消失
  → 已分配的角色-菜单关系可保留，重新购买后自动恢复
```

## 角色分配菜单时的过滤

```
平台管理员创建/编辑平台角色：
  → 只能选 level=PLATFORM 的菜单
  → 无 App 维度过滤

租户管理员创建/编辑租户角色：
  → 只能选 level=TENANT 的菜单
  → 且菜单必须属于 (kava-base ∪ 已购App) 的范围
  → 不能分配未购买 App 的菜单
```

## App 购买后的租户定制流程

租户购买 App 后，需要为 App 实例定制品牌信息：

```
① 购买/订阅
   租户在 Kava 后台 → 应用市场 → 选择 App → 付费

② 租户定制
   管理员设置 App 实例的：名称、图标、描述、主题色等

③ 注册 OAuth Client
   为该租户的 App 实例创建 OAuth Client
   Client ID + Secret → App/小程序使用

④ 生效
   租户成员打开定制后的 App/小程序
   OAuth 登录 → 绑定到该租户 → 看到定制后的界面 → 数据隔离
```

同一套 App 源码，不同租户拥有不同 OAuth Client，接入各自隔离的租户数据。对终端用户而言，每个租户的 App 是独立的 App 或小程序。

## 新增 App 的操作流程

```
1. 在 sys_app 中注册 App（code, name, status）

2. 在 sys_menu 中新增菜单（level=PLATFORM 和/或 TENANT）

3. 在 sys_app_menu 中关联 App 和菜单
   - 可复用已有菜单（如会员管理、消息通知）
   - 可创建 App 专属菜单

4. 租户购买后自动生效
```

## 白标 SaaS 模式

本模型本质是**白标 SaaS（White-label SaaS）**：

- 同一套 App 产品，租户换皮后成为自己的 App/小程序
- 平台方统一开发、运维、迭代，租户按需购买
- 数据靠 tenant_id 隔离，OAuth Client 标识不同租户的 App 实例
- 订阅制收费，与持续服务成本匹配

## 与旧设计的对比

| 维度 | 旧设计 | 新设计 |
|------|--------|--------|
| 菜单区分 | scope 单字段（SYSTEM/TENANT/SYSTEM_TENANT） | level + app_id 两个维度 |
| 共享菜单 | SYSTEM_TENANT（语义模糊） | 拆成两条菜单（PLATFORM + TENANT），或 menu 复用 |
| App 关联 | 无 | app_id → sys_app（多对多） |
| 购买机制 | sys_tenant.menu_id 逗号分隔字符串 | sys_tenant_app 关系表 |
| 推送方式 | 创建时手动写入 | 购买即生效，声明式 |
| 扩展性 | 加新 App 要改逻辑 | 加新 App 只需加数据 |

## 待定事项

- [ ] OAuth Client 创建流程（自动 or 手动，当前先手动）
- [ ] 共享模块（如会员管理）是否需要在租户没有任何 App 时隐藏（当前设计为始终可见）
- [ ] App 间数据联动策略（当前不需要，保持 App 独立）
- [ ] 第三方开发者接入 App 的可能性（当前所有 App 由平台方开发）
